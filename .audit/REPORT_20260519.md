## [SEV-2] Home Cache-Control is `private, no-store` ‚Äî defeats the edge cache
**Route:** `/`
**Steps:**
1. Visit `https://yellow-pink.vercel.app/`
2. DevTools ‚Üí Network ‚Üí click the document request ‚Üí Response Headers
**Expected:** `cache-control: public, s-maxage=300, stale-while-revalidate=86400` (per `next.config.ts`)
**Actual:** `cache-control: private, no-cache, no-store, max-age=0, must-revalidate`
**Why it matters:** The `next.config.ts` `headers()` rule for `/` declares a 5-min CDN cache window. Result: every visitor render-roundtrips Supabase. The whole edge-caching commit (`46fd42e`) is no-op on the home page.
**Possible fix:** Drop `force-dynamic` from `src/app/layout.tsx`, fetch the audience-resolution promos via a leaf-component-friendly approach (or accept the audience-mismatch tradeoff for the cache hit). Same finding likely applies to `/shop`, `/blog`, `/product/[slug]` ‚Äî confirm those next.

## [SEV-2] Internal `?cat=` vs canonical `?category=` query-param mismatch ‚Äî duplicate-content trap
**Route:** `/shop?category=Makeup` (header link) ‚Üí silently becomes `/shop?cat=Makeup` (URL after page renders)
**Steps:**
1. From `/`, click the **Makeup** link in the header. `href="/shop?category=Makeup"`.
2. Page loads, but `location.search` is now `?cat=Makeup`.
3. The category-tab strip at the top of the shop page is a mix: "All" tab uses `cat=Makeup#main`, but the per-row chips and the header nav use `category=`.
4. `/sitemap.xml` declares the canonical URL as `?category=‚Ä¶`.
**Expected:** One canonical query-param name across header, in-page sub-tabs, and sitemap.
**Actual:** Two URLs serve the **same** filtered page: `/shop?category=Makeup` (from header + sitemap) and `/shop?cat=Makeup` (from in-page sub-tab + after navigation). Either one returns the same 163-product Makeup grid.
**Why it matters:**
- SEO: Google sees both URLs linked from internal pages ‚Üí both get indexed ‚Üí ranking is split between two identical pages.
- Analytics: `pageviews` are reported under two different paths.
- Sharing: a customer who clicks a chip then copies the URL ends up sharing the `?cat=` one, which isn't the canonical.
**Possible fix:** Pick `?category=` everywhere (it matches the sitemap + nav). Audit `src/sections/collection/CollectionPage.tsx` ‚Äî the in-page sub-tab / chip URL generation is reading/writing `cat`. Either rewrite to `category`, or add a server-side 301 from `?cat=X` ‚Üí `?category=X` in `src/proxy.ts`.

## [SEV-0] `orders` table is publicly readable via anonymous Supabase REST ‚Äî full PII leak
**Probe:** From the browser console on `yellow-pink.vercel.app`, with no admin auth:

```js
  headers: { apikey: ANON, Authorization: `Bearer ${ANON}` }

```
**Actual response (HTTP 200):**
```json
[{
  "id": "5e3b1537-‚Ä¶",
  "order_number": "YP-WSCLB4",
  "email": null,
  "first_name": "QA",
  "last_name": "Test",

  "address": "123 Test Street, Gulshan",
  "city": "Karachi",

  "zip": "75000",

  "user_id": ‚Ä¶, "coupon_code": ‚Ä¶, "discount_amount": ‚Ä¶

```
**Why it matters:** Every order ‚Äî past, present, future ‚Äî is readable by anyone with a web browser. **Full PII leak** (name, email, phone, address, city, postal code, payment method, items, totals, tracking, user_id). Today there's only 1 test order so the practical exposure is tiny, but **this would be a Pakistan-DPP-Bill-level breach the second a real customer places an order**. Same JS one-liner above gives you the entire orders table.
**Possible fix:** Add an RLS policy on `public.orders`:
```sql
ALTER TABLE public.orders ENABLE ROW LEVEL SECURITY;
DROP POLICY IF EXISTS orders_owner_read ON public.orders;
CREATE POLICY orders_owner_read ON public.orders FOR SELECT
  USING (auth.uid() = user_id);
-- (anon access only via existing SECURITY DEFINER RPC `lookup_order(order_number, phone)`)
```
**Then re-run the probe** above and confirm it returns `[]` for anon.

## [SEV-1] `coupons` table is publicly readable ‚Äî all promo codes leaked
**Probe:** Same anon-REST trick, hitting `/rest/v1/coupons?select=code,type,value,active,used_count&limit=200`.
**Actual:** HTTP 200, returns all 3 coupon codes: **`WELCOME10`** (10% off), **`SAVE200`**, **`NEWUSER15`** ‚Äî including their type, value, active status, used count.
**Why it matters:**
- Any visitor can scrape every promo code, including unpublished / segment-targeted ones.
- Competitor automation can list-snipe codes and burn the usage quota.
- "First-order only" codes can be exhausted by a single bad actor.
- Customers can find codes that were meant for specific email-domain segments (e.g. press codes, influencer codes).
**Possible fix:** Add RLS policy on `public.coupons`:
```sql
ALTER TABLE public.coupons ENABLE ROW LEVEL SECURITY;
-- No SELECT for anon at all ‚Äî coupon validation goes through a
-- SECURITY DEFINER RPC like `validate_coupon(code, cart_subtotal)`
DROP POLICY IF EXISTS coupons_read ON public.coupons;
```
The cart applies coupons via a server action already (`src/app/checkout/rewards-actions.ts` per my read of the repo) ‚Äî that's `SERVICE_ROLE`-keyed and bypasses RLS, so closing anon SELECT shouldn't break the checkout flow.

## [SEV-2] `site_settings` table is publicly readable ‚Äî minor key-leak surface
**Probe:** Same anon-REST trick, hitting `/rest/v1/site_settings?select=key`.
**Actual:** HTTP 200, returns all 38 keys. Most are intentionally public (hero copy, store address, loyalty multipliers, payment-method toggles).

## [SEV-2] `og:title` meta contains raw user query (e.g. `<img onerror=alert(1)>`)
**Route:** `/shop?q=%22%3E%3Cimg+src%3Dx+onerror%3Dalert%281%29%3E`

**Possible fix:** When building `metadata.openGraph.title` from the search query, run through `escapeHtml()` (or use `String(query).replace(/[<>"&']/g, ‚Ä¶)`). Probably in the `generateMetadata` of `src/app/shop/page.tsx`.
